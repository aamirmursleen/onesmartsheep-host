MamaSign Reseller Blueprint

BuyWhiteLabel jaisa reseller system MamaSign ke liye kaise banega

Ye report sirf reseller side ke workflow, backend logic, database, APIs, security aur implementation phases par focus karti hai. Super Admin already ban chuka hai, isliye ab target hai reseller dashboard ko BuyWhiteLabel style banana.

Prepared for MamaSign - focus: reseller system, white-label flow, SaaS functionality. Date: May 10, 2026.

1. Core Idea Simple Words Me

BuyWhiteLabel me reseller apna SaaS brand create karta hai. Wo domain connect karta hai, logo/colors lagata hai, SMTP set karta hai, plans banata hai, clients onboard karta hai, aur un clients ko apne brand ke under SaaS use karwata hai.

MamaSign me same logic hoga:

  • Super Admin reseller create karega.
  • Reseller apna branded MamaSign portal setup karega.
  • Reseller clients ko invite karega ya public signup link dega.
  • Clients documents send/sign/manage karenge.
  • Reseller apni pricing/plans control karega.
  • Super Admin ko har reseller ka usage, clients aur revenue dikhna chahiye.

6. White-Label Branding Logic

Har request par system hostname read karega aur tenant resolve karega.

Request Host: app.yakesign.com
Find domain in tenant_domains
tenant_id = yake_sign_tenant
Load branding from tenant_branding
Load SMTP from tenant_smtp
Load plans/features from tenant plans
Render app with YakeSign branding

Iska matlab same MamaSign codebase multiple brands serve karega.

9. Billing Options

Option A - Central MamaSign Billing

Client payments MamaSign Stripe me aati hain. Super Admin reseller revenue/commission calculate karta hai.

Best for first version kyun ke easy hai.

Option B - Reseller Own Stripe

Reseller apna Stripe connect karta hai. Client payment direct reseller ke Stripe me jati hai.

Best for full white-label lekin complex hai.

Recommendation: Phase 1 me central MamaSign billing rakho. Phase 2 me Stripe Connect ya reseller-owned Stripe add karo.

13. Security Rules

  • Har query me `tenant_id` required ho.
  • Client data `client_company_id` ke bina access na ho.
  • SMTP password encrypted store ho.
  • API keys hashed store hon, plain text nahi.
  • Reseller doosre reseller ka domain claim na kar sake.
  • Super Admin impersonation audit logs me save ho.
  • Domain verification ke bina custom domain active na ho.
  • Billing webhook signature verify ho.
  • Plan limits server-side enforce hon, sirf frontend par nahi.

14. Step-by-Step Build Plan

Phase 1 - Reseller Tenant Foundation
Super Admin ke created reseller ko tenant se connect karo. Hostname based tenant resolver banao.
Phase 2 - Reseller Login and Dashboard Shell
Reseller ke liye dashboard routes: overview, branding, domains, SMTP, plans, clients.
Phase 3 - Branding System
Logo, favicon, platform name, support email save/load. Dashboard aur signing emails me apply.
Phase 4 - Custom Domain
Domain add, DNS instructions, CNAME verify, status active/pending/failed.
Phase 5 - SMTP
SMTP credentials encrypted save, test email, tenant email sending fallback.
Phase 6 - Plans and Limits
Reseller plans create kare, features/limits set kare, client assign kare.
Phase 7 - Client Management
Invite client, public signup, suspend, delete, plan change, usage view.
Phase 8 - Billing
Stripe checkout/webhooks, subscription status, invoice logs, revenue dashboard.
Phase 9 - API/Automation
Client API keys, usage limits, webhooks, Pabbly/Zapier/n8n compatibility.
Phase 10 - Analytics
Reseller dashboard metrics: MRR, clients, conversion, document usage, API usage.

15. What We Should Build First

Since Super Admin already exists, first practical step should be: Reseller Tenant Foundation + Reseller Dashboard Shell + Branding Save/Load. Without this foundation, domain, SMTP, plans, clients sab unstable honge.

First Sprint Scope

  • Reseller login route
  • Reseller dashboard layout
  • Tenant resolver
  • Branding settings page
  • Save logo/name/color/support email
  • Load branding on reseller/client portal
  • Basic clients page with empty state
  • Basic plans page with empty state

16. Final Recommendation

BuyWhiteLabel ka exact functional pattern MamaSign ke liye ye hai: reseller apna branded SaaS control kare, clients onboard kare, plans banaye, domain/SMTP connect kare, aur MamaSign features ko apne naam se sell kare.

Implementation me sab se important cheez tenant isolation hai. Agar tenant isolation weak hui to ek reseller ka document/client/payment doosre reseller ko leak ho sakta hai. Isliye pehle backend tenant model solid banao, phir features add karo.

Best first build: reseller dashboard + branding + tenant resolver. Uske baad domain, SMTP, clients, plans, billing, API.
